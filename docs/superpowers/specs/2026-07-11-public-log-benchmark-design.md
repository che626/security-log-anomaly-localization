# Public Log Benchmark Design

## Status and decision record

**Status:** Approved design; implementation has not started.

**Owner goal:** Turn the existing ISCC security-log competition work into a credible data-science and machine-learning portfolio project. The primary audience is a data-science or machine-learning internship interviewer. Security logging is the differentiating application domain, not a claim that the project is a production security product.

The project must describe the competition history honestly. The final official ranking was 277; the system-log task B-list score was 0.87111. The repository must not present this as an award. Existing historical OOF and locked-decoder figures remain clearly marked as private-competition evidence, not public-benchmark results.

## Success criteria

The work succeeds when the repository tells a coherent, reproducible story:

1. It evaluates the existing approach on public, documented log datasets without redistributing raw data.
2. It compares the neural approach against credible interpretable and unsupervised baselines under the same split protocol.
3. It reports random, chronological, and template-isolated evaluations where each is meaningful, making leakage risks visible rather than hiding them.
4. It demonstrates both standard anomaly detection and anomaly localisation, then measures cross-system transfer in a second phase.
5. It provides a recruiter-friendly README, a technical report, reproducible commands, result figures, error examples, and concise interview narratives.

Success is **not** defined as beating a particular published number. Every reported result must be generated by this repository under a stated configuration.

## Scope and phases

### Phase 1: core resume-ready benchmark

Datasets:

- **HDFS (Loghub):** canonical sequence-level binary anomaly detection.
- **OpenStack (Loghub):** second public logging system for an independently documented binary anomaly-detection evaluation.

Deliverables:

- unified public-dataset adapters and manifests;
- four baseline families and the existing neural architecture;
- dataset-internal comparison tables and figures;
- random, time, and template-isolated split reports where the dataset permits them;
- a concise public benchmark report and README summary.

This phase answers: *can a private competition solution become a transparent, repeatable machine-learning project?*

### Phase 2: localisation and cross-system generalisation

Datasets:

- **BGL (Loghub):** line-level anomaly labels transformed into contiguous anomaly spans in a documented windowing protocol.
- **Thunderbird (Loghub):** a fixed, chronological, hash-recorded subset rather than the full approximately 30 GB corpus.

Deliverables:

- binary span-localisation evaluations on BGL;
- zero-shot, few-shot fine-tuning, and in-domain Thunderbird experiments;
- transfer curves, calibration plots, resource measurements, and localisation error cases.

This phase answers: *when the logging system changes, how much target data is needed before the approach becomes useful?*

## Public-data policy

Loghub data remain external to the Git repository. The repository will include source URLs, the required citation and licence notes, download instructions, expected checksums, preparation parameters, and generated manifests. It will not include raw Loghub logs, competition CSV files, participant identifiers, cached features, checkpoint files, detailed public predictions, or private competition predictions.

Each preparation manifest records the source revision or download date, input SHA256 values, row/group counts, the selected subset rules, task profile, split seed, and preprocessing configuration. A manifest is the audit record for any result table.

Tiny, fully synthetic fixtures may be committed only for automated tests. They must not contain real raw log records.

## Unified task contract

The model must never invent labels unavailable in a source dataset. Each dataset adapter declares exactly one task profile for each experiment:

| Profile | Input and target | Required outputs | Type head |
| --- | --- | --- | --- |
| `sequence_binary` | Ordered log group with one normal/anomalous label | anomaly probability and binary decision | disabled |
| `span_binary` | Ordered log window with one or more binary anomalous spans | anomaly probability plus inclusive spans | disabled |
| `span_multiclass` | Ordered log window with labelled anomaly categories | anomaly probability, spans, and category | enabled only with real source labels |

HDFS and OpenStack run as `sequence_binary`. BGL and the Thunderbird subset run as `span_binary`. ISCC's ten competition-specific anomaly classes are never mapped onto public data by hand.

For BGL and Thunderbird, an adapter converts official abnormal line labels into contiguous inclusive spans within fixed windows. Window length, stride, boundary handling, span merging, and treatment of multi-span windows are configuration values recorded in the manifest. Windows are assigned to a split before feature fitting; no overlapping window may place the same original log line in different split partitions.

## Model strategy

The project retains the existing `LogBoundaryNetwork` rather than replacing it with an unrelated model. Its deterministic normalisation, hashed word/bigram/character features, line-level EmbeddingBag layer, CNN plus BiGRU encoder, BIO head, start/end heads, global head, and constrained decoder are portfolio evidence worth preserving.

Task profiles control which components participate:

- `sequence_binary`: train and assess a binary global anomaly decision; localisation heads and the type head are excluded from loss and scoring.
- `span_binary`: collapse labels to normal versus anomalous spans; use binary BIO/boundary signals and the constrained decoder; the multi-class type head is excluded.
- `span_multiclass`: preserve all compatible heads only when an external dataset provides trustworthy category labels.

The implementation must preserve backward compatibility for the private ISCC workflow. Public profiles, adapters, and configuration must be isolated from the existing competition path.

## Baselines and fairness

Every public task uses the same samples, split assignments, and train-only feature fitting for all models.

| Baseline family | Purpose |
| --- | --- |
| Template-frequency / rarity rule | transparent operational baseline |
| TF-IDF plus logistic regression | standard supervised linear baseline |
| Template-statistics plus decision tree | interpretable nonlinear supervised baseline |
| Isolation Forest | label-independent anomaly baseline |
| Existing neural model | proposed representation and sequence-aware approach |

For span tasks, each baseline must expose a documented rule that converts its line or window score to a binary span. Its threshold is chosen from validation data only. Baseline implementations may be simplified, but their aggregation, training features, and threshold policy must be documented so that comparisons remain fair.

## Evaluation protocol

### Splits

Random splits are a diagnostic reference, not the main claim. The main evidence uses chronological and template-isolated splits when timestamps and templates permit them.

- **Random split:** stratified by the dataset's supported label unit, with a fixed seed.
- **Time split:** validation and test observations occur after training observations; group boundaries prevent a session or window leaking across partitions.
- **Template-isolated split:** normalized-template groups are assigned to one partition only, testing behaviour on previously unseen templates.

The exact partition count and label balance must be reported for every manifest. If a dataset lacks a meaningful timestamp or template grouping, the report must say which split is unavailable and why; it must not substitute a different split under the same name.

### Metrics

Sequence-level experiments report precision, recall, F1, PR-AUC, ROC-AUC where valid, false-positive rate, confusion matrix, calibration summary, inference throughput, and peak observed memory where practical.

Span-level experiments additionally report token/line precision, recall, F1, span precision, recall, F1, inclusive IoU, exact-boundary accuracy, and at least five qualitative error examples selected by a documented rule rather than cherry-picking.

All thresholds, decoder parameters, early-stopping choices, and temperature-calibration parameters are selected using validation data only. The final test result is run once per fixed configuration. Retuning after viewing the test score requires a new experiment ID and a separate result row.

### Required experiment matrix

| Experiment block | Dataset(s) | Comparison | Question answered |
| --- | --- | --- | --- |
| Standard detection | HDFS, OpenStack | four baselines versus neural model | Is the sequence-aware approach competitive with common methods? |
| Leakage robustness | Phase 1 data | random versus time versus template-isolated splits | Does performance depend on memorising time-adjacent records or templates? |
| Localisation ablation | BGL | global-only, localisation-only, and full compatible multi-head setup | Do localisation signals improve localisation quality? |
| Cross-system transfer | BGL to Thunderbird subset | zero-shot, few-shot fine-tune, full target training | What target-data cost is required for adaptation? |
| Confidence reliability | all principal neural runs | uncalibrated versus validation-calibrated predictions | Are scores usable for thresholded decisions? |

The localisation ablation compares compatible binary heads only; it does not compare the unavailable ISCC type classifier on public binary labels.

## Reproducibility and repository deliverables

The public workflow will be expressed as a small number of documented stages:

1. obtain a licensed dataset locally;
2. verify it and produce a manifest;
3. prepare the configured task profile and split;
4. train a named baseline or neural configuration;
5. evaluate once, generating a metrics record and figures;
6. build the report from aggregate result files.

Each run records the experiment ID, dataset manifest ID, profile, code revision, dependency version, random seed, device, elapsed time, peak memory when available, and selected validation settings.

The repository landing page will contain:

- a plain-language 30-second project overview;
- truthful ISCC context separated from public benchmark results;
- architecture and data-flow diagrams;
- a compact headline table linking to the full report;
- a reproducibility quick start and data-policy note;
- limitations, especially dataset shift and the lack of public fine-grained anomaly types.

The technical report will include exploratory data analysis, feature and label protocol, baselines, split rationale, complete metrics, calibration, error analysis, compute cost, and threats to validity. The existing Streamlit app may receive only lightweight presentation changes: it can display synthetic/anonymous examples and aggregate public metrics, but it must not become a separate product or host raw datasets.

## Testing and error handling

Automated tests must cover:

1. adapter schema validation, label alignment, and deterministic manifest generation;
2. window-to-source-line mappings and multi-span boundary cases;
3. split disjointness for original source lines, groups, and normalized templates as appropriate;
4. train-only fitting for feature statistics, thresholds, decoder settings, and calibration;
5. task-profile masking so disabled heads cannot affect loss, metrics, or output;
6. metric calculations on normal-only, anomaly-only, empty-span, and boundary-adjacent fixtures;
7. report aggregation rejecting mixed manifests or incompatible task profiles.

Failures must be explicit: missing raw files, checksum mismatches, unsupported labels, split leakage, invalid span order, and incompatible checkpoints should halt the run with actionable messages rather than silently continuing. Continuous integration runs only synthetic fast tests, formatting, and documentation checks; it never downloads a public corpus or trains a large model.

## Career-use package

After real results exist, add three evidence-backed materials:

- **Resume bullets:** method, dataset scope, evaluation discipline, and real result; no unverified number or award wording.
- **30-second explanation:** a contest prototype was rebuilt into a public reproducible anomaly-detection benchmark, then tested for transfer across log systems.
- **Interview deep dive:** explain class imbalance, leakage controls, the difference between detection and localisation, the baseline choice, and an error case that motivated a limitation or future improvement.

## Explicit exclusions

- No claim of an ISCC award or top ranking.
- No raw-data redistribution or private-data publication.
- No manually fabricated public anomaly classes.
- No test-set threshold tuning or selective reporting of only the best split.
- No unrelated dashboard, web platform, deployment service, or production-security claim.
